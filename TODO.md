## Высокий приоритет

### Structured tool calling (Mistral function calling)

Сейчас все LLM-ответы парсятся вручную: классификация типа запроса, извлечение товаров, список нужных позиций — всё через `.strip().lower()` и regex. Это хрупко: модель может ответить «Это задача» вместо «задача», и парсинг сломается. Mistral поддерживает [function calling](https://docs.mistral.ai/capabilities/function_calling/) — объявляем функции со схемой аргументов, модель возвращает структурированный JSON.

**Что изменить:**

- Классификацию `consultation / task` вынести в инструмент `classify_intent(type: "consultation" | "task")`
- Извлечение товаров из запроса — `extract_products(items: string[])`
- Определение списка нужных товаров — `list_required_products(products: string[])`
- Фильтр релевантности — `filter_relevant(ids: int[], reason: string)`
- Убрать ручной парсинг, добавить Pydantic-валидацию аргументов

**Почему важно:** явные контракты упрощают отладку, тестирование (можно мокировать ответы по схеме) и будущую смену модели.

## Средний приоритет

### Eval-харнесс

Сейчас качество системы проверяется вручную: запустил, написал несколько сообщений, «кажется работает». Это не воспроизводимо и не позволяет регрессировать.

**Что нужно:**

- Golden dataset: 40–60 размеченных запросов. Каждый пример — `{query, expected_intent, expected_products[], scenario}`. Сценарии: консультация, задача с уточнением, задача без уточнения, follow-up, off-topic.
- Скрипт `evals/run_evals.py`: прогоняет каждый запрос через полный pipeline, сравнивает результат с эталоном
- Метрики:
  - `intent_accuracy` — доля правильно классифицированных запросов
  - `product_recall@5` — доля ожидаемых товаров в top-5 результате
  - `offtopic_precision` — доля корректно отклонённых нерелевантных запросов
  - `clarification_rate` — как часто задаётся уточняющий вопрос там, где надо
- Вывод: таблица с метриками + список «провалившихся» примеров с diff
- Запускать локально перед каждым значимым изменением промптов или pipeline

### Гибридный поиск (BM25 + Embeddings)

BM25 — лексический поиск: если пользователь пишет «краска для стен» а в каталоге «интерьерная эмаль», совпадения не будет. Семантический поиск решает эту проблему.

**Подход:**

- Mistral Embeddings API (`mistral-embed`) для векторизации товаров при старте
- Хранить векторы в памяти (numpy array или hnswlib-индекс)
- При запросе: параллельно BM25 + embeddings, объединять через Reciprocal Rank Fusion (RRF)
- Параметр `alpha` для настройки веса: `score = alpha * bm25 + (1 - alpha) * semantic`
- Мерить качество через eval-харнесс (метрика `product_recall@5`)

**Осторожности:**

- Ре-индексация при изменении каталога (сейчас каталог статичен — не проблема)
- Стоимость: 1 вызов embed при старте на ~1000 товаров, затем только для запросов

### Multi-step agent loop

Если BM25 ничего не нашёл — сейчас возвращается пустой ответ. Агент мог бы переформулировать запрос и повторить.

**Схема:**

```
поиск → пусто → LLM переформулирует запрос → поиск повторно → ...
```

**Детали:**

- Максимум 2 итерации перефомулирования (stop condition — счётчик)
- Стратегии переформулирования: синонимы, упрощение (убрать прилагательные), расширение (добавить категорию)
- Если после 2 итераций всё равно пусто — честно ответить что ничего не нашли
- Логировать каждую итерацию: исходный запрос → переформулировка → результат

**Что изменить:**

- Mistral поддерживает SSE streaming (`stream=True`)
- FastAPI endpoint `/api/chat` переходит на `StreamingResponse`
- Стримить только «длинные» вызовы: экспертный ответ и список товаров
- JS-виджет: отображать токены по мере поступления (append в DOM)
- Карточки товаров отправлять отдельным событием после завершения стрима

**Эффект:** субъективная скорость ответа улучшается значительно, даже если общее время не меняется.
